# Overview

# Changed
## `RandAugment` `random_state` handling [gh#19](https://github.com/imaug/imaug/pull/19)
The class `RandAugment` handled the parameter `random_state` inconsistently for
its parent class `meta.Sequential`. This resulted in assertion errors once it
was set. Instead of overwriting `seed` and keeping the provided `random_state`,
the new version resets `random_state` to `'depreciated'` once `seed` is
overwritten. Additionally, a new test case is added to check the new behavior.

## Removed end-of-life Python versions from CI pipeline [cb#9](https://codeberg.org/imaug/imaug/pulls/9)
In order to reduce number of the supported Python versions, the end-of-life versions
`3.8` and `3.9` are dropped from the CI pipeline in favor of newer versions.

## Add Python 3.14 to CI pipeline [gh#20](https://github.com/imaug/imaug/pull/20)
Add Python 3.14 testing to `test_master.yml`, `test_pull_requests.yml`, and `build_wheels.yml`
GitHub Action files.

## `_blend_alpha_uint8_single_alpha_` always returns copies [gh#23](https://github.com/imaug/imaug/pull/23)
The inplace modification of `image_fg` caused errors when applied on non-C-contiguous views.
Thus, the function was changed such that it always returns copies, even when called with `inplace=True`.
It was verfied that calling functions do not rely on inplace modifications for proper functionality.

## Moved image `_downloader.py` from `checks` to `imgaug` [gh#21](https://github.com/imaug/imaug/pull/21)
For creating the example images for the documentation it is necessary to download online sample images
similar to the workflow for the checks. Instead of copy the `download` function for the documentation, 
the function was moved to serve as a general utility function for the library.

## Add `beautifulsoup4` as explicit dependency for building the documentation [gh#21](https://github.com/imaug/imaug/pull/21)
The dependency `dashtable.html2rst` depends on `beautifulsoup4` as optional dependency.

## Switch from bash to python for collecting example image generation scripts [gh#21](https://github.com/imaug/imaug/pull/21)
Replace `gen_all.sh` with `generate_all.py` for better cross platform compatibility.

## Use readthedocs images for README file instead of github files [gh#21](https://github.com/imaug/imaug/pull/21)
Use the images in RTD for github README.md file for smaller repository size.

## Use PyPi trusted publisher from GitHub Actions [cb#10](https://codeberg.org/imaug/imaug/pulls/10)
Improve security of release process by removing the dependency on long-lived API tokes. Additionally automate releases.
Any push to the `master` branch will from now on trigger a distributions build which will be uploaded to TestPyPi.
If the push contains a git tag, a new version will be uploaded to PyPi.

## Handling of non-c-contiguous views in `_blend_alpha_uint8_single_alpha_`[#23](https://github.com/imaug/imaug/pull/23)
The calling functions of `_blend_alpha_uint8_single_alpha_` do not always guarantee C-contiguous arrays base 
arrays (i.e. which own their data). This leads to problems when the function sets the `inplace`
parameter to `True`. In this case the `_normalize_cv2_input_arr_` makes sure that `image_fg` is a C-contiguous
base array when used as CV2 input, but does not change it when used as CV2 destination. This fix makes sure that
also the destination array matches the CV2 requirements, while sacrificing the (real) inplace modification of 
`image_fg`.

## Fix incorrect dtype docstrings [gh#21](https://github.com/imaug/imaug/pull/21)
The script `collect_dtype_support.py` is very sensitive to incorrectly defined dtype docstrings.
Pull request [#640](https://github.com/aleju/imgaug/pull/640) introduced a circular look-up definition
in `imgaug.augmenters.segmentation`for the class `Voronoi`. Another problem was a misplacement of the
version added docstring in `imgaug.augmenters.arithmetrics` for the function `cutout` in
[#598](https://github.com/aleju/imgaug/pull/598).

All download data come from Wikipedia. It became a little stricter with respect to the

## Limit supported opencv-python version [cb#3](https://codeberg.org/imaug/imaug/pulls/3)
`opencv-python` has a bug in `warpAffine` which requires a fix upstream. At the moment it
looks like 4.14 will include a [fix](https://github.com/opencv/opencv-python/issues/1207).
Without restricting the supported image dtypes the corresponding tests will fail. Therefore
a restriction with respect to the supported opencv-python version to **<4.13** was introduced in
the install requirements.

## Limit max supported numpy version [cb#3](https://codeberg.org/imaug/imaug/pulls/3)
`numpy` **2.4** [removed](https://numpy.org/doc/stable/release/2.4.0-notes.html#raise-typeerror-on-attempt-to-convert-array-with-ndim-0-to-scalar)
the support to convert an higher dimensional array to scalar. Yet, this behavior is used throughout this
libaray which requires additional code changes. In order to provide support for Python 3.14 without waiting
for the required fixes, the install requirements limit support to numpy **<2.4** for now. We are confident
, that we can support higher `numpy versions in future releases.

## Mitigate NumPy 2.3 removal of out-of-bound indexes on empty arrays [cb#6](https://codeberg.org/imaug/imaug/pulls/6)
NumPy 2.3 [removed](https://numpy.org/doc/stable/release/2.3.0-notes.html#expired-deprecations) the previously [depreciated](https://numpy.org/doc/stable/release/1.20.0-notes.html#indexing-errors-will-be-reported-even-when-index-result-is-empty) out-of-bound indexes on empty arrays. This caused test failures for zero-sized array axes in `meta.py`. To continue support for the given test set, mitigations
were set in place.

## Use fixed `imagecorruptions-imaug` version 1.1.5 [cb#8](https://codeberg.org/imaug/imaug/pulls/8)
The minimal supported `imagecorruptions-imaug` version was increased from `1.1.3` to `1.1.5`.
Version `1.1.4` removed the dependency on the unmaintained `pkg_resources` package. Unfortunately the mitigation code 
introduced a logic error, which made version `1.1.5` necessary.

# Improved

## Handling of views in `_multiply_elementwise_to_uint8_`[gh#19](https://github.com/imaug/imaug/pull/19)
The elementwise multiplication with `cv2.multiply` breaks if a
RGB image is sliced in the channel dimension and used as destination.
The error only seems to occur if the view's base shape has a singular first
dimension. Therefore a true copy of the image is used as destination instead.

## Enable dtype support matrix script on Windows [gh#21](https://github.com/imaug/imaug/pull/21)
The original `collect_dtype_support.py` script hard-coded the front-slash directory separator and
assumed a root suffix. Replaced it with the OS specific separator and make script more robust with respect
to relative directory definitions.

## Fix tempfile issue for readme image generation script[gh#21](https://github.com/imaug/imaug/pull/21)
According to https://stackoverflow.com/a/54768241 Windows struggles with `NamedTemporaryFile` without
the additional parameter `delete_on_close=False`.
